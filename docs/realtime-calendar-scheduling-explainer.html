Realtime Calendar Scheduling
Feature Explainer

This flow replaces hardcoded slots with live Google Calendar availability while preserving business rules (counselor routing, blocked hours, day windows, and slot duration). Booking is conflict-safe and updates the counselor calendar instantly.

Source of truth: Google Calendar
Runtime: Netlify Functions
Availability API + Booking API
Race condition protection on submit

End-to-End Flow

1) User picks date

QualifiedLeadForm sends date + lead category to availability endpoint.

2) Netlify builds candidate slots

Rules applied from counselor config: 1-hour slots, blocked 2 PM, day windows, 60-minute buffer.

3) Google FreeBusy check

FreeBusy API returns busy windows for counselor calendar. Overlapping candidate slots are removed.

4) UI shows live slots

Remaining slots are rendered as selectable options in the same existing UI.

5) Submit creates event

Booking endpoint rechecks availability and inserts event. Slot becomes busy for subsequent users.

Netlify Functions and Their Purpose

  • _counselorConfig.ts: leadCategory -> counselor mapping + scheduling policy.
  • _gcal.ts: Google JWT auth + timezone/date helpers + overlap utilities.
  • _slotEngine.ts: builds candidate slots and filters by FreeBusy response.
  • gcal-availability.ts: returns live available slots for selected date.
  • gcal-booking.ts: rechecks slot, creates calendar event, handles slot conflicts.

Google Calendar APIs Used

  • FreeBusy API: read busy blocks to compute real-time availability.
  • Events Insert API: create booking event on counselor calendar at submit.
  • Calendar sharing with service account gives read/write access to counselor calendars.
  • Booking conflict path: if slot is no longer available at submit, API returns 409 SLOT_TAKEN.

Read path = FreeBusy. Write path = Events Insert.

API Contracts in This Feature

POST/.netlify/functions/gcal-availability

Input: { leadCategory, date }
Output: { slots: [{ label, startIso, endIso }] }

POST/.netlify/functions/gcal-booking

Input: selected slot + parent/student details.
Output: { eventId, htmlLink } or 409 SLOT_TAKEN.

What Happens on Submit in QualifiedLeadForm

Frontend submit sequence

  • Validate form fields (date, slot, parent name, email).
  • Map selected slot label to exact startIso/endIso.
  • Call booking endpoint to reserve slot first.
  • If booking succeeds, continue existing final form submission flow.

Failure modes

  • Slot conflict: returns 409, refresh slots, ask user to reselect.
  • Booking API error: frontend blocks final submit to prevent inconsistent state.
  • Success: event created and slot immediately becomes unavailable for others.
